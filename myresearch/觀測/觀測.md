# 觀測（Observability）研究結果

> 更新：2026-02-26  
> 核心問題：觀測是上線才用的嗎？可以融入開發工作流嗎？

---

## 先釐清問題

你問的「觀測」其實包含三個不同的東西，需要分開討論：

```
1. Code Metrics（程式碼品質指標）
   → 靜態分析，不需要執行程式
   → 開發時就能用，完全不需要上線

2. Agent Observability（AI agent 本身的可觀測性）
   → 追蹤 review agent 的決策過程
   → 開發工作流就能用

3. Application Observability（應用程式執行時的可觀測性）
   → 延遲、錯誤率、traces
   → 傳統上是上線才用，但 2025 年趨勢是「Shift Left」
   → 可以在 staging/local 模擬
```

這三個的成熟度和適用時機完全不同。

---

## 2025 年的關鍵趨勢：Shift Left Observability

傳統觀念：觀測 = 上線後監控  
2025 年趨勢：觀測 = 從寫程式碼開始就要有

> "Observability is becoming a design-time concern rather than a runtime patch."  
> — Hydrolix, Observability in 2025

**Shift Left 的具體做法：**
- Pre-merge instrumentation testing：確保 telemetry 輸出在部署前就是正確的
- Local observability sandboxes：在本地模擬 production 條件
- Observability as code：dashboard、alerts、metrics 都版本控制

## Part 1：Code Metrics（最容易上手，開發時就能用）

### 什麼是 Code Metrics

不是 runtime 指標，是靜態分析程式碼結構的指標。

| 指標 | 意義 | 高值代表 |
|------|------|----------|
| Cyclomatic Complexity | 程式碼中獨立路徑的數量 | 難以測試、容易有 bug |
| Cognitive Complexity | 人類理解程式碼的難度 | 難以維護 |
| Lines of Code (LOC) | 函數/檔案的大小 | 可能違反單一職責 |
| Duplication % | 重複程式碼比例 | 技術債 |
| Dependency Depth | 模組依賴深度 | 耦合度高 |

**重要：這些指標不需要執行程式，CI 就能跑。**

### 和 Review Agent 的關係

Code metrics 可以作為 semantic review 的**前置過濾器**：

```
Cyclomatic Complexity > 10 的函數
  → 自動標記為「高風險，需要更仔細的 semantic review」
  → 同時要求更高的 mutation score（因為邏輯複雜）

Cognitive Complexity > 15 的函數
  → 標記為「可讀性問題，建議重構」
  → 不一定是 bug，但是維護風險
```

### 這個專案怎麼加

**Option A：ESLint（最輕量）**

```javascript
// .eslintrc.js 加入
rules: {
  'complexity': ['warn', { max: 10 }],  // Cyclomatic complexity
  'max-lines-per-function': ['warn', { max: 50 }],
  'max-depth': ['warn', { max: 4 }],
}
```

**Option B：SonarQube / SonarCloud（完整）**

```yaml
# .github/workflows/sonar.yml
- name: SonarCloud Scan
  uses: SonarSource/sonarcloud-github-action@master
  env:
    SONAR_TOKEN: ${{ secrets.SONAR_TOKEN }}
```

SonarCloud 免費支援 public repo，private repo 有免費額度。  
提供 Cognitive Complexity、Duplication、Security Hotspots 的 dashboard。

**這個專案的高風險函數（預測）：**
- `lib/api/report-processing.ts` — 報告生成邏輯複雜
- `lib/api/line-webhook.ts` — 多種 event type 的 switch 邏輯
- `lib/utils/week.ts` — 台灣曆法週期計算有邊界條件

## Part 2：Agent Observability（Review Agent 本身的可觀測性）

### 為什麼 Review Agent 需要被觀測

你的 review agent 本身就是一個 AI 系統，它也會：
- 做出錯誤判斷（false positive / false negative）
- 在某些類型的程式碼上表現差
- 隨著 LLM 版本更新而行為改變
- 消耗不同的 token 和時間

如果不觀測 agent 本身，你不知道它是否在正確工作。

### 要追蹤什麼

```
每次 review 的 trace：
  - 輸入：diff 大小、context 大小（tokens）
  - 輸出：findings 數量、severity 分布
  - 效能：LLM 呼叫延遲、總耗時
  - 品質：findings 被開發者接受/拒絕的比例（需要人工標記）
  - 成本：token 用量、費用
```

### 工具選擇

**Langfuse（推薦，開源可自架）**

```typescript
// 在 review agent 中加入 tracing
import { Langfuse } from 'langfuse'

const langfuse = new Langfuse({
  publicKey: process.env.LANGFUSE_PUBLIC_KEY,
  secretKey: process.env.LANGFUSE_SECRET_KEY,
})

async function semanticReview(diff: string) {
  const trace = langfuse.trace({ name: 'semantic-review' })
  
  const span = trace.span({ name: 'llm-review', input: { diff } })
  const result = await llm.review(context, prompt)
  span.end({ output: result, usage: { tokens: result.usage } })
  
  return result
}
```

**這樣你就能看到：**
- 每次 review 的完整 trace
- 哪些 PR 觸發了哪些 findings
- LLM 的 token 用量趨勢
- 哪些 findings 被開發者接受（需要手動標記）

### 開發階段就能用

Agent observability 不需要上線。你在本地跑 review agent 時，trace 就會被記錄。

這讓你在開發 review agent 本身時，能看到它的行為是否符合預期。

## Part 3：Application Observability（應用程式的可觀測性）

### 傳統定義

追蹤應用程式執行時的三個訊號：

```
Logs    → 發生了什麼事（這個專案已有 Pino）
Metrics → 系統健康狀態（延遲、錯誤率、吞吐量）
Traces  → 一個 request 經過哪些服務（分散式追蹤）
```

### 這部分確實偏向上線

你的直覺是對的。Application observability 的核心價值在於：
- 真實用戶的行為模式
- 真實的負載下的效能
- 真實的錯誤率

這些在開發環境很難完全模擬。

### 但可以「Shift Left」的部分

**1. 本地模擬（Local Observability Sandbox）**

```bash
# 用 Docker Compose 在本地跑完整的 observability stack
# Jaeger（traces）+ Prometheus（metrics）+ Grafana（dashboard）
docker-compose up jaeger prometheus grafana
```

然後在本地跑 load test，觀察 traces 和 metrics。

**2. Staging 環境的 Synthetic Monitoring**

在 staging 環境用腳本模擬真實用戶行為：
```
每 5 分鐘：模擬教師登入 → 建立觀察記錄 → 上傳照片
每小時：模擬園長查看報告 → 觸發 PDF 生成
```

這樣即使沒有真實用戶，也能持續觀測系統行為。

**3. Pre-merge Instrumentation Testing**

確保新程式碼的 telemetry 輸出是正確的：
```typescript
// 測試：確保 API 有正確發出 metrics
it('emits latency metric on observation create', async () => {
  const metrics = captureMetrics()
  await observationsApi.create(...)
  expect(metrics).toContain({ name: 'api.observation.create.duration' })
})
```

### 這個專案的 Application Observability 基礎

這個專案已有 **Pino** 結構化 logging，這是很好的起點。

**現有基礎：**
```typescript
// lib/utils/logger.ts 已存在
// 已有結構化 log，可以直接接 log aggregation
```

**可以加的（不需要大改動）：**

```typescript
// 在 API routes 加入基本 metrics
// app/api/observations/route.ts

const start = Date.now()
try {
  const result = await handler(req)
  logger.info({ 
    duration: Date.now() - start,
    status: 200,
    route: '/api/observations'
  })
  return result
} catch (error) {
  logger.error({ 
    duration: Date.now() - start,
    status: 500,
    route: '/api/observations',
    error: error.message
  })
}
```

**Supabase 本身的可觀測性：**
- Supabase Dashboard 有 Query Performance 分析
- 可以看慢查詢、RLS policy 的執行時間
- 不需要額外設定

## 整合視角：三層觀測在開發工作流的位置

```
開發階段（Development）
  ├── Code Metrics（靜態分析）
  │     → ESLint complexity rules
  │     → SonarCloud scan in CI
  │     → 輸出：複雜度報告，標記高風險函數
  │
  └── Agent Observability（Review Agent 的 trace）
        → Langfuse tracing
        → 輸出：每次 review 的決策 trace

測試/Staging 階段（Pre-production）
  ├── Instrumentation Testing
  │     → 確保 telemetry 輸出正確
  │
  └── Synthetic Monitoring
        → 模擬用戶行為，觀測系統反應
        → 可以發現效能問題（Supabase 慢查詢等）

上線後（Production）
  └── Application Observability（完整）
        → Pino logs → log aggregation（Datadog/Grafana）
        → Supabase Query Performance
        → Error tracking（Sentry）
```

## 觀測和 Review Agent 的結合點

這是你問的核心：**觀測能融入 review agent 嗎？**

可以，有兩個方向：

**方向 1：Code Metrics 作為 Review Agent 的輸入**

```
PR diff
  + Code Metrics（complexity、duplication）
  + Mutation Score（如果有跑）
  ↓
Review Agent 的 context

→ LLM 知道「這個函數 complexity = 15，mutation score = 45%」
→ 可以做出更有根據的判斷：「這個函數邏輯複雜且測試不足，建議重構」
```

**方向 2：Review Agent 的 findings 作為觀測資料**

```
每次 review 的 findings → Langfuse trace
  → 追蹤：哪些類型的問題最常出現？
  → 追蹤：哪些 findings 被開發者接受？
  → 追蹤：review agent 的準確率趨勢

這些資料可以用來改進 review agent 本身（prompt 調整、context 優化）
```

## Part 5：Code Metrics 實作詳解

### 什麼是 Cyclomatic Complexity（循環複雜度）

每個 `if`、`else if`、`while`、`for`、`case`、`&&`、`||`、`??` 都讓複雜度 +1，起始值是 1。

```typescript
// complexity = 1（只有一條路）
function getOrgId(user: User) {
  return user.org_id
}

// complexity = 4（1 + 3個決策點）
function canEdit(user: User, obs: Observation) {
  if (user.role === 'principal') return true      // +1
  if (obs.created_by === user.id) return true     // +1
  return user.org_id === obs.org_id               // +1（&&）
}
```

**判斷標準：**
| 複雜度 | 意義 |
|--------|------|
| 1–5 | 簡單，容易測試 |
| 6–10 | 中等，可接受 |
| 11–20 | 高，需要注意 |
| > 20 | 危險，應該重構 |

---

### 這個專案的分析結果（2026-02-26）

用簡易腳本對 `lib/api/` 和 `lib/utils/` 做檔案層級的複雜度掃描：

| 狀態 | 檔案 | 複雜度 | 行數 |
|------|------|--------|------|
| 🔴 | lib/api/reports.ts | 76 | 592 |
| 🔴 | lib/api/analytics.ts | 65 | 513 |
| 🔴 | lib/api/line-webhook.ts | 60 | 329 |
| 🔴 | lib/api/observations.ts | 57 | 414 |
| 🔴 | lib/api/parent-reports.ts | 54 | 315 |
| 🔴 | lib/api/report-processing.ts | 51 | 201 |
| 🔴 | lib/api/report-sender.ts | 50 | 207 |
| 🔴 | lib/utils/error-handler.ts | 48 | 191 |
| 🔴 | lib/api/export.ts | 45 | 320 |
| 🔴 | lib/api/dashboard.ts | 41 | 206 |

> **注意：** 這是檔案層級的累計值，不是單一函數的複雜度。一個 500 行的檔案有 50+ 複雜度是正常的，重點是找出**單一函數**複雜度過高的情況。

**解讀：**
- `reports.ts`（76, 592行）和 `analytics.ts`（65, 513行）是最需要關注的
- `line-webhook.ts`（60, 329行）— 行數相對少但複雜度高，代表函數內部邏輯密集
- `observations.ts`（57, 414行）— 核心業務邏輯，也是多租戶隔離最關鍵的地方

**這些高複雜度檔案和 Mutation Testing 的關係：**
複雜度高 → 邏輯分支多 → 需要更多測試才能覆蓋 → Mutation Score 更可能偏低

### 怎麼在這個專案正式設定

**Step 1：ESLint 加入 complexity rules（最快）**

```javascript
// eslint.config.mjs 加入
{
  rules: {
    'complexity': ['warn', { max: 10 }],        // 單一函數 cyclomatic complexity
    'max-lines-per-function': ['warn', { max: 60, skipBlankLines: true }],
    'max-depth': ['warn', { max: 4 }],          // 巢狀深度
    'max-params': ['warn', { max: 5 }],         // 函數參數數量
  }
}
```

跑 `pnpm lint` 就會看到哪些函數超標。

**Step 2：SonarCloud（完整 dashboard，免費）**

```yaml
# .github/workflows/sonar.yml
name: SonarCloud
on: [push, pull_request]
jobs:
  sonar:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
        with: { fetch-depth: 0 }
      - uses: SonarSource/sonarcloud-github-action@master
        env:
          SONAR_TOKEN: ${{ secrets.SONAR_TOKEN }}
```

```properties
# sonar-project.properties
sonar.projectKey=your-project-key
sonar.organization=your-org
sonar.sources=app,components,lib,hooks
sonar.exclusions=**/*.test.ts,**/*.spec.ts,**/node_modules/**
sonar.javascript.lcov.reportPaths=coverage/lcov.info
```

SonarCloud 提供：
- 每個函數的 Cognitive Complexity
- 重複程式碼百分比
- Security Hotspots
- Quality Gate（可設定 PR 不通過條件）

### C# 專案怎麼做

C# 有三個主要方式，從輕到重：

**方式 1：Visual Studio 內建（最簡單）**

```
Analyze → Calculate Code Metrics for Solution
```

直接產出報告，包含：
- Cyclomatic Complexity（每個方法）
- Maintainability Index（0-100，越高越好）
- Depth of Inheritance
- Class Coupling
- Lines of Code

**方式 2：dotnet CLI + Roslyn Analyzers（CI 用）**

```xml
<!-- .csproj 加入 -->
<PackageReference Include="Microsoft.CodeAnalysis.NetAnalyzers" Version="*" />
<PackageReference Include="SonarAnalyzer.CSharp" Version="*" />
```

```bash
# 建置時自動分析
dotnet build --configuration Release /p:RunAnalyzersDuringBuild=true

# 或用 dotnet format 檢查
dotnet format --verify-no-changes
```

**方式 3：SonarQube / SonarCloud（和 TypeScript 一樣）**

```xml
<!-- Directory.Build.props -->
<PropertyGroup>
  <SonarQubeExclude>false</SonarQubeExclude>
</PropertyGroup>
```

```bash
dotnet sonarscanner begin /k:"project-key" /o:"org" /d:sonar.token="$SONAR_TOKEN"
dotnet build
dotnet sonarscanner end /d:sonar.token="$SONAR_TOKEN"
```

**C# 特有的工具：**

| 工具 | 特點 |
|------|------|
| NDepend | 最強大的 .NET 靜態分析，有 CQL 查詢語言，付費 |
| ReSharper | JetBrains，IDE 整合，即時顯示複雜度，付費 |
| Roslyn Analyzers | 免費，Microsoft 官方，整合在 build pipeline |
| SonarQube C# plugin | 免費開源版，和 TypeScript 版功能相同 |

**C# Cyclomatic Complexity 的計算和 TypeScript 一樣：**
```csharp
// complexity = 1
public string GetName() => user.Name;

// complexity = 4
public bool CanEdit(User user, Observation obs) {
    if (user.Role == "principal") return true;   // +1
    if (obs.CreatedBy == user.Id) return true;   // +1
    return user.OrgId == obs.OrgId;              // +1（比較運算）
}
```

**C# 的 Maintainability Index（MI）是額外的指標：**
```
MI = 171 - 5.2 * ln(HalsteadVolume) - 0.23 * CyclomaticComplexity - 16.2 * ln(LinesOfCode)

> 85：容易維護（綠）
20–85：中等（黃）
< 20：難以維護（紅）
```

Visual Studio 直接顯示這個數字，不需要手動計算。
