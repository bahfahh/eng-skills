## 為什麼需要 ExecPlan

AI agent 長時間執行時，context window 會壓縮，**會遺忘之前做過的事、決策的理由、下一步該做什麼**。ExecPlan 就是 agent 的「外部記憶體」——重讀就能恢復完整狀態並繼續執行。

---

## 何時使用

符合以下**任一**條件時，使用 ExecPlan 取代預設的 4 份文件流程：

- 預計執行跨越多天或多個工作階段
- 跨 3 個以上模組的系統變更
- 需要原型驗證才能確認可行性
- 未來可能由不熟悉專案的人接手
- 任務中途可能暫停，之後需從文件恢復上下文

不符合 → 使用預設流程（requirements / plan / todolist / acceptance / tests）。

## 核心心法

**ExecPlan ≠ 設計文件** → 是「可執行規格書」。
**ExecPlan ≠ 給作者看** → 是給「完全陌生的人或 agent」看。

ExecPlan 永遠可以：
- **從頭讀**：新手拿到就能執行到底
- **從中斷點繼續**：看 Progress 就知道現在在哪、下一步是什麼
- **單獨存在也能完成任務**：不依賴外部文件、先前對話、任何記憶

一句總結：**讓任何人只拿這一份檔案，就能把事情做完，而且跑得出結果。**

## 三種操作模式

### 撰寫 ExecPlan 時（Authoring）

從下方骨架開始，邊研究邊充實內容。徹底閱讀（並重讀）相關原始碼，產出準確的規格。不要先寫空殼再補——研究到哪，寫到哪。

### 執行 ExecPlan 時（Implementing）

- **不詢問使用者「下一步做什麼」**，直接進入下一個 milestone
- 每個停止點更新所有 section，在 Progress 中明確記錄已完成的進度與下一步
- 部分完成的項目拆成「已完成」與「剩餘」兩條
- 自主解決歧義，將決策記錄在 Decision Log
- 頻繁 commit

### 討論 ExecPlan 時（Discussing）

將決策記錄在 Decision Log，確保任何對規格的修改都有明確的原因。ExecPlan 是 living document，任何時刻都必須能只從這份文件重新開始。

### 研究階段（Researching）

遇到高難度需求或重大未知時，用 milestone 實作 proof of concept 或 toy implementation 來驗證可行性。主動閱讀函式庫原始碼、深入研究，將原型納入計畫以指導後續完整實作。

## 不可違反的約束

1. **完全自包含**：只靠這份文件 + working tree 就能執行。不寫「如前所述」「參見架構文件」，所有必要資訊直接嵌入文件
2. **新手可執行**：術語首次出現時立即定義，並指出它在 repo 中對應的檔案或指令
3. **產出可運作的行為**：不是「符合定義的程式碼變更」，而是可示範的、跑得出結果的行為
4. **Living Document**：隨進度、發現、決策持續修訂，每次修訂後仍必須完全自包含

## 撰寫指南

### 目的與意圖優先

開頭用幾句話說明：從使用者角度，這個變更讓人能做到什麼之前做不到的事，以及如何看到它在運作。然後引導讀者完成確切步驟——要編輯什麼、要執行什麼、應該觀察到什麼。

### 散文優先

用完整句子描述。避免 checklist、表格、長列舉，除非簡潔反而會模糊意思。Checklist 只用在 Progress section。

### 自包含與白話

引入「daemon」「middleware」「filter graph」等術語時，立即用白話定義，並指出它在 repo 中對應的檔案或指令。不寫「如前所述」「根據架構文件」，即使重複也要把資訊寫在這裡。

引用已 check-in 的先前 ExecPlan 時，寫出完整路徑。未 check-in 的，把相關內容直接納入。

### 避免常見失敗模式

- **不依賴未定義的術語**
- **不把功能描述得太窄**，導致程式碼能編譯但實際上什麼都沒做
- **不把關鍵決策丟給讀者**——有歧義時，在計畫中解決並說明為什麼選這條路
- **過度解釋使用者可見的效果，少描述實作細節**

### 明確指定 Repo 上下文

- 檔案使用 repo-relative 完整路徑
- 精確指名函式與模組
- 新檔案要說明建立位置
- 跨多個區域時，先寫一段導覽說明各部分如何關聯
- 執行指令時，標明工作目錄與完整指令
- 結果依賴環境時，說明假設條件並提供替代方案

### 可觀察的驗收

驗收描述人可驗證的行為，不描述內部屬性：

- 正確：「啟動 server 後，GET /health 回傳 HTTP 200，body 為 OK」
- 正確：「執行測試，新測試 test_health 在變更前失敗、變更後通過」
- 錯誤：「新增了 HealthCheck struct」

內部變更也要說明如何示範其影響（例如：跑測試、展示使用新行為的情境）。

### 驗證不可省略

包含執行測試的指令、啟動系統的方式、觀察有用行為的方法。附上預期輸出與錯誤訊息，讓新手能分辨成功與失敗。盡可能展示變更的有效性超越「能編譯」——透過小型 E2E 情境、CLI 呼叫、或 HTTP request/response 記錄。

### 冪等與安全

步驟設計為可重複執行不造成損壞或偏移。步驟可能中途失敗時，附上重試或調整方式。需要破壞性操作時，說明備份或安全回退方案。優先使用可逐步驗證的增量變更。

### 捕捉證據

步驟產生的終端輸出、短 diff、log，以縮排範例形式嵌入文件。保持簡潔，聚焦在證明成功的內容。需要 patch 時，優先使用檔案範圍的 diff 或小段摘錄，而非貼大段程式碼。

## Milestone 設計

Milestone 是敘事，不是官僚流程。每個 milestone 用一段散文描述：範圍、完成後什麼東西存在了而之前不存在、要執行的指令、預期觀察到的驗收結果。像故事一樣可讀：目標 → 工作 → 結果 → 證明。

Progress 和 Milestone 是不同的東西：Milestone 講故事，Progress 追蹤細粒度工作。兩者都必須存在。

每個 milestone 必須可獨立驗證，且遞增地實現整體目標。不要為了簡短而省略 milestone 中可能對未來實作至關重要的細節。

## 原型與平行實作

### 原型 Milestone（Prototyping）

當需要降低大型變更的風險時，加入明確的原型 milestone。例如：在依賴中新增低階 operator 驗證可行性，或探索兩種組合順序並測量效果。

- 明確標示範圍為「prototyping」
- 描述如何執行與觀察結果
- 定義升級（promote）或捨棄（discard）的判斷標準
- 對外部函式庫或新功能領域，建立獨立的 spike 驗證可行性，證明外部函式庫如預期運作

保持原型為增量且可測試的。

### 平行實作

遷移期間可保留新舊兩條路徑並行（例如 adapter 與舊路徑共存），以降低風險或讓測試在大型遷移中持續通過。描述如何驗證兩條路徑、如何安全退役舊路徑並保持測試通過。

優先使用增量新增，再做減法，確保測試持續通過。

## Living Document 四個必要 Section

ExecPlan 必須包含並持續維護以下四個 section，這是 agent context 壓縮後的「外部記憶體」：

### Progress

用 checkbox 追蹤細粒度步驟，含時間戳。每個停止點都必須記錄，即使需要把部分完成的項目拆成兩條。此 section 必須永遠反映工作的實際當前狀態。

### Surprises & Discoveries

發現非預期的 optimizer 行為、效能取捨、bug、或影響方法的語意時，記錄觀察與簡短證據（測試輸出為佳）。

### Decision Log

每個決策記錄決策內容、理由、日期。中途改變方向時，在此說明原因，並在 Progress 中反映影響。

### Outcomes & Retrospective

在主要 milestone 或全部完成時，總結成果、剩餘工作、學到的教訓。將結果與原始目的對照。

### 修訂規則

- 每次修訂後，文件仍必須完全自包含
- 修訂必須反映到所有相關 section（不只改一處）
- 在文件底部附註修訂內容與原因
- ExecPlan 不只描述 what，還要描述 why

## ExecPlan 骨架

```markdown
# <動作導向的簡短標題>

本文件為 Living Document。Progress / Surprises & Discoveries / Decision Log /
Outcomes & Retrospective 必須隨工作進展持續更新。

## Purpose / Big Picture

用幾句話說明：這個變更讓使用者能做到什麼之前做不到的事，
以及如何觀察到它在運作。描述你將啟用的使用者可見行為。

## Progress

用 checkbox 追蹤細粒度步驟。每個停止點都必須記錄。
用時間戳衡量進度速率。

- [x] (2025-10-01 13:00Z) 已完成的步驟描述
- [ ] 未完成的步驟描述
- [ ] 部分完成（已完成：X；剩餘：Y）

## Surprises & Discoveries

記錄實作中發現的非預期行為，附簡短證據。

- Observation: …
  Evidence: …

## Decision Log

記錄每個決策、理由、日期。

- Decision: …
  Rationale: …
  Date: …

## Outcomes & Retrospective

在主要 milestone 或完成時，總結成果、差距、教訓。
將結果與原始目的對照。

## Context and Orientation

描述與本任務相關的現狀，假設讀者一無所知。
列出關鍵檔案與模組的完整路徑。定義所有非顯而易見的術語。
不引用先前計畫。

## Plan of Work

以散文描述編輯與新增的順序。每個編輯指明檔案完整路徑、
函式/模組位置、要插入或修改的內容。保持具體且精簡。

## Milestones

每個 milestone 以散文描述：範圍、完成後新增了什麼、
執行指令、驗收行為。像故事一樣可讀：目標 → 工作 → 結果 → 證明。

## Concrete Steps

列出確切的執行指令與工作目錄。指令產生輸出時，
附上預期的簡短輸出範例供比對。隨工作進展更新。

## Validation and Acceptance

描述如何啟動或操作系統、觀察什麼結果。
驗收以行為描述，附具體輸入與輸出。
例：「執行 <測試指令>，預期 N 個通過；新測試 <name> 在變更前失敗、變更後通過」

## Idempotence and Recovery

說明哪些步驟可安全重複。風險步驟附重試或回滾路徑。
完成後保持環境乾淨。

## Artifacts and Notes

嵌入最重要的終端輸出、diff、程式碼片段（縮排格式）。
保持簡潔，聚焦在證明成功的內容。

## Interfaces and Dependencies

指明使用的函式庫、模組、服務及原因。
列出 milestone 完成後必須存在的型別、介面、函式簽名，使用完整路徑。

例：在 crates/foo/planner.rs 中定義：

    pub trait Planner {
        fn plan(&self, observed: &Observed) -> Vec<Action>;
    }
```

## 與預設流程的差異

| 面向 | 預設流程 | ExecPlan |
|------|---------|----------|
| 文件數量 | 4 份分散文件 | 單一自包含文件 |
| 讀者假設 | 熟悉專案的開發者 | 完全陌生的人或 agent |
| 核心問題 | 「怎麼做」 | 「怎麼讓任何人做完並跑出結果」 |
| 更新頻率 | 階段性 | 每個停止點 |
| 執行模式 | 階段 Gate 驗證後推進 | 自主推進，頻繁 commit |
| 歧義處理 | 提問使用者 | 自主決策 + 記錄 Decision Log |
| Context 遺失時 | 需重新對話釐清 | 重讀文件即可恢復 |

**注意**：Step 1（需求確認）仍然適用。ExecPlan 取代的是 Step 2-5 的文件與流程。
