# Review Agent 研究結果

> 更新：2026-02-26  
> 背景：專案已有 rule-based review（steering 規則檢查）。本文件聚焦在超越規則的兩個方向：**Semantic LLM Review** 和 **Mutation Testing**。

---

## 核心問題釐清

### Rule-based 的根本限制

Steering rules 是**事後文件**——bug 發生 → 補規則。所以：

```
Rule-based review 能抓到的：已知的已知錯誤
Semantic review 能抓到的：邏輯錯誤、意圖偏差、未文件化的邊界問題
Mutation testing 能抓到的：測試覆蓋的盲點（測試通過但沒真正驗證行為）
```

這三層互補，不是替代關係。

## Part 1：Semantic LLM Review

### 1.1 「語意」從哪裡來？（最關鍵的問題）

這是設計 semantic review agent 最容易搞錯的地方。

**錯誤理解：** 語意 = 更聰明的 prompt，讓 LLM 用常識判斷程式碼好壞  
**正確理解：** 語意 = **這段程式碼應該做什麼** 的定義，來自專案的 requirements 和 design docs

```
語意的來源層次（由高到低）：

1. Requirements docs    → 功能應該做什麼（.kiro/specs/）
2. Design docs          → 架構應該長什麼樣（.kiro/specs/design.md）
3. Existing codebase    → 這個專案的慣例和模式
4. PR diff context      → 這次改動的意圖
5. LLM general knowledge → 通用程式設計知識（最弱，最容易 hallucinate）
```

**結論：語意不是靠 prompt 定義的，是靠 context 注入的。**

2025 年這個概念被稱為 **Context Engineering**（Andrej Karpathy、Shopify CEO Tobi Lütke 都在推這個詞）：

> "Context engineering is the art of providing all the context for the task to be plausibly solvable by the LLM."

Prompt 只是指令，Context 才是讓 LLM 能做出正確判斷的資訊基礎。

---

### 1.2 Context Engineering：要餵什麼給 LLM

一個 semantic review agent 的 context 應該包含：

```
[必要]
- PR diff（這次改了什麼）
- 被改動檔案的完整內容（不只是 diff）
- 相關的 requirements 段落（這個功能應該做什麼）

[重要]
- 相關的 design 決策（架構為什麼這樣設計）
- 同類型的現有檔案（這個專案的慣例是什麼）
- 被改動功能的現有測試（預期行為是什麼）

[選擇性]
- 相關的 memory/lessons learned（已知的陷阱）
```

**Context 的品質 > Prompt 的技巧。**

---

### 1.3 針對這個專案的 Context 組裝

這個專案的 semantic source 對應：

| Context 類型 | 來源 |
|-------------|------|
| Requirements | `.kiro/specs/observation-system/requirements.md` |
| Architecture intent | `.kiro/specs/observation-system/design.md` |
| Coding patterns | `.kiro/steering/tech.md` + `code-conventions.md` |
| Known pitfalls | `docs/memory/` 相關檔案 |
| Expected behavior | `tests/` 對應的測試檔案 |

**Context 組裝策略：**

不要把所有文件都塞進去（Context Rot 問題——context 太長 LLM 反而忽略中間內容）。

應該根據 diff 的檔案路徑，**選擇性地**注入相關 context：

```
diff 改動了 lib/api/observations.ts
  → 注入 requirements 中關於 observation 的段落
  → 注入 design.md 中關於 API layer 的段落
  → 注入 hooks/useObservations.ts（呼叫端，了解預期介面）
  → 注入 tests/unit/lib/api/ 下的相關測試
  → 注入 docs/memory/database/ 的 lessons（如果有 DB 相關改動）
```

### 1.4 Prompt 設計

Prompt 的角色是**指定審查的思維框架**，不是定義語意。

**錯誤的 prompt 方向（rule-based 思維）：**
```
"檢查這段程式碼是否符合以下規則：
1. 所有 query 必須帶 organization_id
2. hooks 只能放在 hooks/ 目錄
..."
```
這只是把 steering 搬到 prompt，沒有超越 rule-based。

**正確的 prompt 方向（semantic 思維）：**

```markdown
你是一個資深的 code reviewer，專門審查這個多租戶幼兒園 SaaS 系統。

## 你的審查框架

**1. 意圖對齊（Intent Alignment）**
這段程式碼是否真的實現了 requirements 中描述的功能？
有沒有「程式碼跑得動但做錯事」的情況？

**2. 邊界條件推理（Boundary Reasoning）**
不要只看 happy path。主動思考：
- 如果 organization_id 是 null 會怎樣？
- 如果同時有兩個 request 進來會怎樣？
- 如果 user 在操作中途 logout 會怎樣？

**3. 隱性假設（Hidden Assumptions）**
這段程式碼依賴了哪些沒有明說的假設？
這些假設在什麼情況下會不成立？

**4. 跨層一致性（Cross-layer Consistency）**
這個改動是否和其他層（hook、component、DB schema）保持一致？
有沒有只改了一半的情況？

## 輸出格式

```json
{
  "findings": [
    {
      "severity": "critical|warning|suggestion",
      "type": "intent_mismatch|boundary_issue|hidden_assumption|inconsistency",
      "location": "file:line",
      "description": "具體描述問題",
      "reasoning": "為什麼這是問題（引用 requirements 或 design 的依據）",
      "suggestion": "建議怎麼修"
    }
  ],
  "overall": "pass|needs_review|block"
}
```

**重要：如果你不確定，說不確定。不要為了給出 finding 而 hallucinate。**
```

### 1.5 Multi-Agent Critique（進階）

單一 LLM 有確認偏誤——它傾向認為程式碼是對的。

研究顯示 multi-agent debate 能提升約 11% 的 bug 偵測率。

**最小可行的雙 agent 設計：**

```
Agent A（Reviewer）：
  - 拿到 context + diff
  - 做 semantic review
  - 輸出初步 findings

Agent B（Critic）：
  - 拿到 context + diff + Agent A 的 findings
  - 任務：找 Agent A 漏掉的問題，或質疑 Agent A 的判斷
  - Prompt 重點："假設 Agent A 的 review 是不完整的，你能找到什麼它沒提到的問題？"

最終輸出：合併兩個 agent 的 findings，去重後排序
```

這個設計不需要複雜的 orchestration，兩次 LLM call 就能實現。

### 1.6 這個專案的高價值審查點

根據這個專案的特性，semantic review 最有價值的地方：

**多租戶隔離的語意正確性**
```typescript
// 這段程式碼 rule-based 會通過（有帶 organization_id）
// 但 semantic review 應該發現：students join 沒有 org filter
.from('observations')
.eq('organization_id', orgId)
.select('*, students(*)')  // ← students 沒有 org_id filter，可能跨租戶洩漏
```

**Auth 狀態機的完整性**
```
Requirements 說：pending 用戶只能訪問 reset-password
Semantic review 應該問：
- 如果 pending 用戶直接 POST /api/observations 呢？
- middleware 是否在所有 API routes 都做了 status 檢查？
```

**RLS 和 API 層的雙重保護**
```
Design 說：RLS 是最後防線，API 層也要有 org_id filter
Semantic review 應該問：
- 這個新 API 有沒有在 RLS 之外也做了 org_id 過濾？
- 如果 RLS 被意外 disable，API 層能擋住嗎？
```

### 1.7 工具選擇

**Option A：用現有商業工具（最快）**

| 工具 | 特點 | 適合場景 |
|------|------|----------|
| CodeRabbit | 最廣泛使用，PR 自動 review | 想快速上手 |
| Greptile | 全 repo context，aggressive bug hunting | 需要跨檔案分析 |
| Qodo | 結合 static analysis + LLM | 需要高精確度 |

商業工具的限制：不知道你的 requirements docs，語意來源只有 codebase，沒有 spec。

**Option B：自建（最符合這個專案）**

```typescript
// 概念：scripts/semantic-review.ts
async function semanticReview(prDiff: string) {
  // 1. 根據 diff 找相關的 requirements 段落
  const relevantRequirements = await extractRelevantRequirements(prDiff)
  
  // 2. 找相關的現有程式碼（呼叫端、測試）
  const relatedCode = await findRelatedCode(prDiff)
  
  // 3. 組裝 context
  const context = assembleContext({
    diff: prDiff,
    requirements: relevantRequirements,
    relatedCode,
    designDocs: relevantDesignSections,
  })
  
  // 4. Agent A review
  const reviewA = await llm.review(context, reviewerPrompt)
  
  // 5. Agent B critique
  const reviewB = await llm.critique(context, reviewA, criticPrompt)
  
  // 6. 合併輸出
  return mergeFindings(reviewA, reviewB)
}
```

## Part 2：Mutation Testing

### 2.1 核心概念

Mutation testing 解決的問題：**程式碼覆蓋率說謊。**

```
覆蓋率 80% 的意思：80% 的程式碼被執行到了
Mutation score 80% 的意思：80% 的邏輯錯誤會被測試抓到

這是完全不同的事。
```

**運作方式：**
```
原始程式碼 → 自動產生「突變版本」→ 對每個突變跑測試
  突變被測試抓到（測試失敗）= Mutant Killed ✅
  突變沒被抓到（測試通過）= Mutant Survived ❌ → 測試盲點
```

**突變的種類（Stryker 會自動產生）：**
```typescript
// 原始
if (org_id === user.org_id)

// 突變 1：條件反轉
if (org_id !== user.org_id)

// 突變 2：移除條件
if (true)

// 突變 3：邊界值
if (org_id >= user.org_id)
```

如果這些突變都沒有讓測試失敗，代表你的測試根本沒有驗證這個邏輯。

### 2.2 Stryker 設定（TypeScript + Jest）

這個專案用 pnpm + Jest，設定如下：

**安裝：**
```bash
pnpm add -D @stryker-mutator/core @stryker-mutator/jest-runner @stryker-mutator/typescript-checker
```

**設定檔 `stryker.config.mjs`：**
```javascript
// @ts-check
/** @type {import('@stryker-mutator/api/core').PartialStrykerOptions} */
const config = {
  testRunner: 'jest',
  jest: {
    projectType: 'custom',
    configFile: 'jest.config.js',
  },
  checkers: ['typescript'],
  tsconfigFile: 'tsconfig.json',
  
  // 先從最重要的模組開始，不要一次跑整個專案（太慢）
  mutate: [
    'lib/api/observations.ts',
    'lib/api/reports.ts',
    'lib/utils/week.ts',
    'lib/utils/permissions.ts',
    // 逐步擴展
  ],
  
  reporters: ['html', 'clear-text', 'json'],
  htmlReporter: { fileName: 'reports/mutation/index.html' },
  
  // 避免跑太久
  timeoutMS: 10000,
  concurrency: 2,
}

export default config
```

**執行：**
```bash
# 跑 mutation testing
npx stryker run

# 只跑特定檔案
npx stryker run --mutate "lib/api/observations.ts"
```

### 2.3 分數解讀

| Mutation Score | 代表意義 |
|---------------|----------|
| < 60% | 測試嚴重不足，大量邏輯沒有被驗證 |
| 60–80% | 有基本覆蓋，但有明顯盲點 |
| 80–90% | 良好，還有改進空間 |
| > 90% | 強健的測試套件 |

### 2.4 Surviving Mutant 的處理流程

```
Stryker 報告 surviving mutant
  ↓
確認這個突變是否代表真實的 bug 風險
  ↓
  ├─ 是 → 補寫測試，明確驗證這個行為
  └─ 否（例如：純 UI 文字）→ 在 stryker.config 中 exclude 這個 mutant
```

**範例：Surviving Mutant 報告**
```
Mutant survived: lib/api/observations.ts: line 45
Mutator: ConditionalExpression
- if (data.organization_id === orgId)
+ if (true)

→ 代表：你的測試從來沒有驗證「organization_id 不符合時應該被拒絕」
→ 應該補：it('rejects observation from different org', ...)
```

## Part 3：兩者如何搭配

```
開發流程中的位置：

[寫程式碼]
    ↓
[Layer 1: Static Gate]     ← 已有：TypeScript + ESLint + Tests
    ↓ pass
[Layer 2: Semantic Review] ← 新增：LLM 審查意圖對齊和邊界問題
    ↓ pass
[Merge]
    ↓
[定期：Mutation Testing]   ← 不在 PR gate，而是定期跑（每週或每個 milestone）
    ↓
[補測試 → 提升 mutation score]
```

**為什麼 Mutation Testing 不放在 PR gate？**

因為 Stryker 跑完整個專案需要幾分鐘到幾十分鐘，放在 PR gate 會讓開發流程太慢。

正確的做法是：
- PR gate：只跑被改動檔案的 mutation（`--mutate` 指定）
- 定期（每週 CI）：跑完整的 mutation report，追蹤 score 趨勢

## Part 4：實作優先順序建議

**Phase 1（1-2 天）：Mutation Testing 基礎**
- 安裝 Stryker
- 先跑 `lib/utils/` 下的 pure functions（最快看到結果）
- 建立 baseline mutation score

**Phase 2（3-5 天）：Semantic Review 基礎版**
- 整理 requirements.md 中的關鍵段落，做成 LLM 可讀的 context 片段
- 寫一個簡單的 script：拿 git diff → 組裝 context → 呼叫 LLM → 輸出 findings
- 先手動跑，不用自動化

**Phase 3（之後）：整合和自動化**
- 把 semantic review 接進 GitHub Actions
- 加入 Agent B（Critic）
- 把 mutation testing 加進週期性 CI

## 附錄：Context Rot 的注意事項

Context 太長會讓 LLM 忽略中間的內容（Lost in the Middle 問題）。

**實際限制：**
- 把最重要的 context 放在開頭和結尾
- 單次 review 的 context 建議不超過 30K tokens
- 如果 diff 很大，分批 review（按檔案或功能模組）

**這個專案的 context 估算：**
```
requirements.md 相關段落：~2K tokens
design.md 相關段落：~2K tokens  
PR diff（一般大小）：~3-5K tokens
相關現有程式碼：~5-10K tokens
總計：~12-19K tokens → 在合理範圍內
```
